# AI 纪念站审阅态重构设计

> 适用范围：`/sites/[siteId]` 后台详情页的页面定位、信息架构、交互分层与审阅态预览体验。
>

> - [AI 纪念站跨相册章节生成与详情轻编辑设计](/Users/user/Documents/codes/work/docs/superpowers/specs/2026-05-29-memory-site-cross-album-actions-design.md)

## 背景

当前 `\`/sites/[siteId]\`` 的后台详情页，本质上还是“摘要卡 + 操作侧栏”：

1. 左侧只展示封面图、标题、引子和少量章节摘要。
2. 右侧承载 `换一版选图`、`替换某一张图`、`修改文案`、`发布纪念站` 等动作。
3. 用户无法在进入页面的第一时间看到“这份纪念站作为成品到底成立不成立”。

这导致后台详情页虽然已经具备真实动作能力，但整体感知仍然像一个信息不足的后台卡片页，而不是“审阅一份接近可交付成品的纪念站草稿”。

这和 V1 已经明确的产品承诺不一致。V1 售卖的不是“功能按钮”，而是“像摄影师交付件的单页成品”。因此后台详情页的首要职责应该从“展示一份摘要”改成“展示一份完整成品预览”。

## 目标

1. 让用户进入 `\`/sites/[siteId]\`` 后，第一眼先看到完整纪念站预览，而不是后台摘要。
2. 让后台详情页和公开纪念页在结构上尽量一致，减少“后台看到的”和“发布出去的”割裂。
3. 保留 V1 的轻编辑边界，但把编辑动作降级为辅助能力，不破坏成品阅读感。
4. 让 `换一版选图`、`替换某一张图`、`修改文案` 的结果能立即反馈在同一页预览里。

## 非目标

1. 不把详情页改造成自由排版编辑器。
2. 不在正文中直接暴露逐段 hover toolbar、拖拽手柄或富文本编辑框。
3. 不新增模板切换、模块增删或章节自由排序。
4. 不重写公开纪念页的核心视觉语言。

## 核心判断

### 1. 详情页的产品定位改为“审阅态成品页”

`/sites/[siteId]` 不再是“后台详情卡片页”，而是“带轻工具条的成品预览页”。

用户心智应该是：

1. 我先在这里看成品是否成立。
2. 如果不满意，再用轻量动作修一下。
3. 修完之后继续留在当前页审阅结果。
4. 最后从这里发布，或跳去看公开页。

### 2. 编辑能力必须退到正文之外

一旦把按钮、输入框、虚线占位框直接塞进纪念站正文，页面立刻会从“成品”退化成“编辑器”。这会削弱纪念站的付费感和交付感。

因此本次重构要求：

1. 正文始终尽量保持公开页的阅读感。
2. 编辑入口集中在顶部工具条与其展开的轻面板里。
3. 真正需要输入的操作通过浮层完成，而不是原地把正文改造成表单。

## 页面结构

### 1. 总体分层

页面采用两层结构：

1. 顶部 `sticky` 轻工具条。
2. 下方完整长页预览。

页面不再使用“左预览、右侧栏”的双栏布局，也不再把正文压缩在卡片容器里。

### 2. 顶部轻工具条

工具条职责是“给当前预览页一个低存在感的审阅控制层”，而不是二次做一个后台 header。

工具条包含三类信息：

1. 左侧弱导航：
   - `返回纪念站列表`
   - 当前状态 `草稿 / 已发布`
2. 中部保持留白，不重复堆叠大标题。
3. 右侧主动作：
   - `发布纪念站` 或 `查看公开页`
   - `编辑`

交互规则：

1. 工具条固定在顶部滚动可见。
2. 背景半透明，带轻微模糊和细描边，不使用厚重纯色 bar。
3. 滚动时只做轻微阴影和背景增强，不能压过 hero。

### 2.1 审阅态与全局偏好控件的关系

`focus review layout` 进入纪念站详情后，不应继续叠加 dashboard 通用的全局偏好控件。

原因有两点：

1. 纪念站详情页顶部已经有自己的审阅工具条，再保留右上角语言 / 主题切换，会和工具条主动作抢同一块视觉高地。
2. 纪念站详情页的首要目标是“先看成品”，而不是“先操作后台壳层设置”。

因此规则明确为：

1. `/sites/[siteId]` 进入 `focus review layout` 后，默认隐藏 dashboard 壳层中的 `LocaleSwitcher` 与 `ThemeToggle`。
2. 页面顶部只保留纪念站自己的轻工具条，不再出现第二套右上角浮动按钮。
3. 这条规则仅针对审阅态预览类页面，不要求把 dashboard 其他列表页、设置页的偏好控件一并移除。

### 2.2 页面跳转策略

`/sites` 到 `/sites/[siteId]` 属于同一条后台审阅链路，不应该强制新开窗口。

规则如下：

1. 从纪念站列表进入某个纪念站详情时，保持当前标签页内跳转。
2. 用户在详情页审阅、轻编辑、返回列表，应当形成一个连续的站内流。
3. 只有 `查看公开页` 这种“去看最终对外成品”的动作，才在新标签页打开，避免打断当前后台审阅上下文。

### 2.3 Focus Review Layout

当前问题的根因不是“同标签页还是新窗口”，而是后台常规壳层里的左侧导航与内容宽度分配，会显著压缩纪念站预览区，导致详情页即使改成成品预览，也仍然不够沉浸。

因此 `\`/sites/[siteId]\`` 不应继续停留在标准 dashboard 内容布局里，而应切换到一个专门的 `focus review layout`。

这个能力必须设计成可复用的页面级布局能力，而不是只给纪念站详情硬编码一个例外。后续凡是“需要优先看成品、画布、预览、长文审阅、分享卡预览”的页面，都可以选择进入该 layout。

`focus review layout` 的通用规则：

1. 保持当前标签页内跳转，不额外打开后台新窗口。
2. 页面级收窄、折叠或隐藏左侧大导航占位，把可用宽度优先让给主内容区。
3. 顶部只保留必要的轻量控制层，不再复用密集型 dashboard 双栏壳。
4. 内容区默认按“沉浸式预览优先”分配宽度，而不是按表格 / 表单页的常规密度分配。
5. 布局切换应该由页面声明，而不是每个页面自己重复实现一套隐藏 sidebar 的逻辑。

对纪念站详情页的落地要求：

1. `\`/sites\`` 仍保留现有 dashboard 列表页壳。
2. `\`/sites/[siteId]\`` 进入 `focus review layout`。
3. 进入详情后，正文宽度应接近公开纪念页，避免被左侧 sidebar 长期挤压。
4. 页面顶部只保留本设计定义的轻工具条。
5. `focus review layout` 下不再渲染通用 `PreferenceDock`，避免与审阅工具条重叠。

### 3. 编辑展开轻面板

点击 `编辑` 后，在工具条下方展开一个轻面板，而不是打开右侧抽屉或跳入独立编辑页。

轻面板职责是“选择要改什么”，不承担完整输入过程。

面板内放三类动作：

1. `换一版选图`
   - 直接执行。
   - 执行完成后保留当前滚动位置，并刷新预览。
2. `替换某一张图`
   - 打开定向替换浮层。
3. `修改文案`
   - 打开文案编辑浮层。

交互规则：

1. 轻面板默认收起。
2. 展开后不遮挡正文主体，只占据工具条下方一小段高度。
3. 点击空白区域或再次点击 `编辑` 可收起。
4. 同一时刻只允许一个更深层的浮层处于打开状态。

## 预览区设计

### 1. 预览区尽量直接复用公开页结构

后台详情页的正文应尽量复用公开页的页面骨架：

1. `hero`
2. `section list`
3. `footer`

这意味着用户在后台看到的，应当非常接近发布后的真实观感，而不是一个信息删减版摘要。

### 2. 审阅态与公开页的差异

审阅态只保留必要差异：

1. 多一条顶部轻工具条。
2. 未发布时显示 `草稿` 状态。
3. 已发布时主动作从 `发布纪念站` 切为 `查看公开页`。
4. 预览中的所有变化以当前草稿为准，不要求先发布才能预览。

除此之外，正文排版、视觉节奏、模块顺序都应尽量与公开页一致。

### 3. 正文中不嵌编辑控件

以下内容不应直接出现在正文里：

1. 浮动按钮组。
2. 大面积虚线框与编辑边界。
3. hover 出现的段内操作条。
4. 输入框替换正文文本。

如果需要提示当前在审阅态，提示应放在顶部工具条，而不是写进正文。

### 4. 正文滚动动效

纪念站正文需要有低存在感的滚动进入动效，但它服务的是“阅读节奏”，不是“展示模板技巧”。

#### 4.1 动效目标

1. 让用户在向下滚动时感到内容被逐段揭示，而不是整页静态堆叠。
2. 让图片与文案卡片有轻微生命感，但不抢照片和文字本身的戏。
3. 保持纪念站成品感，避免做成重交互 landing page。

#### 4.2 动效层级

采用“章节级进入 + 章节内错峰揭示”的两层结构：

1. 每个 `section` 作为整体，在首次进入视口时做一次轻量 `fade + rise`。
2. 章节内部再分层进入：
   - 主图先进入；
   - 标题与摘要第二拍进入；
   - 下方 narrative 小卡片做短间隔 stagger。
3. `footer` 复用同一类 reveal 节奏，作为整页阅读的收束。
4. `hero` 保持静稳，不叠加明显的滚动位移、缩放或视差。

#### 4.3 动效强度边界

推荐强度对应本轮确认的 `B` 方案，即“分层揭示”：

1. 位移幅度控制在 `16-24px`。
2. 单层时长控制在 `0.45-0.65s`。
3. 章节内 stagger 间隔控制在 `80-120ms`。
4. 缓动保持柔和，不使用弹跳、明显 overshoot 或夸张 blur。

明确不做的效果：

1. 不做大幅视差。
2. 不做横向漂移。
3. 不做强缩放镜头推进。
4. 不做正文逐行文字动画。

#### 4.4 触发规则

1. 每个章节 reveal 仅在首次进入视口时触发一次，不随用户来回滚动反复播放。
2. 首屏已在视口中的模块不应因初始隐藏造成闪烁或延迟可读。
3. 工具条、编辑面板、成功/失败提示沿用自己的轻反馈，不纳入正文滚动 reveal 系统。

#### 4.5 可访问性

1. 遵守 `prefers-reduced-motion`。
2. 在 reduced motion 模式下，降级为近乎无位移或无动画的静态呈现。
3. 动效不应成为理解页面结构的前提条件，即使完全关闭动画，阅读顺序也必须成立。

## 三类动作的最小交互

### 1. 换一版选图

这是唯一允许直接执行的动作。

执行流程：

1. 用户展开 `编辑` 轻面板。
2. 点击 `换一版选图`。
3. 面板内按钮进入短暂 loading。
4. 请求完成后，正文预览直接更新。
5. 在工具条下方显示轻量成功提示，例如“已切换为另一版选图”。

约束：

1. 不跳页。
2. 不弹确认框。
3. 尽量保持滚动位置稳定。

### 2. 替换某一张图

这是定向替换动作，必须进入浮层。

浮层流程：

1. 先选章节。
2. 再选当前已展示图片。
3. 再选同章节内的候选替换图。
4. 确认提交后关闭浮层，并刷新正文预览。

浮层要求：

1. 明确告诉用户“替换只发生在同章节内”。
2. 如果没有更多候选图，直接在浮层里给出空状态。
3. 不允许跨章节替换。
4. 章节选择、当前图片选择、候选替换图选择统一使用 [`velvet-select.tsx`](/Users/user/Documents/codes/work/src/components/forms/velvet-select.tsx) 提供的选择器，不使用浏览器原生 `select`。
5. 当前图片与候选替换图的选择器，不应直接把整段 narrative 原文塞进触发器主行。
6. 触发器应采用“双层信息”：
   - 主行使用短标签，例如 `图片 1`、`图片 2` 或等价的短标题。
   - 次级信息使用一行较轻的摘要，概括当前图片内容，例如“黄昏雪地里的木屋与栅栏”。
7. 下拉面板可以展示更完整的描述，但关闭状态下的控件宽度应稳定，不能因长文案把整个浮层横向撑满。

### 3. 修改文案

这是文本微调动作，也必须进入浮层。

浮层包含：

1. 站点标题。
2. 副标题。
3. 引子。
4. 结尾。
5. 每个章节的标题与摘要。

交互要求：

1. 保持简单表单，不做富文本。
2. 保存成功后关闭浮层，并立即更新正文预览。
3. 如果用户尚未保存即关闭，按现有产品习惯决定是否需要二次确认；V1 可以先不加复杂离开保护。
4. 若文案编辑浮层里存在章节切换等下拉选择，同样统一使用 [`velvet-select.tsx`](/Users/user/Documents/codes/work/src/components/forms/velvet-select.tsx)。

## 状态与反馈

### 1. 页面状态

页面至少要显式处理以下状态：

1. `草稿`
2. `已发布`
3. `操作中`
4. `操作失败`

### 2. 成功反馈

成功反馈不应使用打断式 toast 堆满页面，建议采用：

1. 工具条下的轻量内联提示。
2. 操作按钮自身 loading / success 反馈。

### 3. 失败反馈

失败反馈优先就近展示：

1. 直接执行动作失败时，在轻面板里显示错误。
2. 浮层动作失败时，在浮层中显示错误。
3. 不要把失败提示丢到正文底部。

## 响应式规则

### 1. 桌面端

1. 工具条保持单行。
2. 轻面板在工具条下横向展开。
3. 正文使用完整长页阅读宽度。

### 2. 移动端

1. 工具条允许换行，但主动作必须始终在首屏可达。
2. `编辑` 的展开层可转为更紧凑的折叠面板。
3. `替换图片` 与 `修改文案` 继续使用全屏 drawer / sheet，避免小 modal 难用。

## 与现有实现的关系

这次重构不否定现有四个动作的服务端能力，而是重构它们在页面中的承载方式：

1. `PATCH` action 能力继续复用。
2. 现有 `replacePhoto`、`editCopy` 的表单逻辑可迁移到新浮层中。
3. 详情页左侧摘要卡不再保留为主要呈现方式。
4. 公开页的 `hero / section / footer` 组件应成为后台预览的第一复用来源。
5. 现有页面中任何原生 `select` 方案，在新审阅态实现里都应收敛到 `VelvetSelect`，保证与 Velvet Plum 语言一致。
6. 纪念站详情页应作为首个接入 `focus review layout` 的页面，但实现方式必须对其他页面可复用。
7. `focus review layout` 目前已有通用偏好按钮承载能力，但纪念站详情页必须显式退出该能力，避免与审阅工具条冲突。
8. 正文 reveal 动效应以通用小型封装接入，优先服务 `hero / section / footer` 复用组件，而不是为单页写一套不可复用的特效脚本。

## 验证标准

当下面四件事同时成立时，可认为这次重构达标：

1. 用户进入详情页后，第一屏看到的是一份完整纪念站成品感预览。
2. 用户可以在不离开当前页的情况下完成轻编辑并看到结果变化。
3. 页面整体仍然更像“审阅成品”，而不是“进入编辑器”。
4. 后台预览与公开页视觉骨架基本一致，不再出现明显割裂。
5. 详情页不再被标准 dashboard sidebar 明显压缩，主预览区宽度足以支撑沉浸式阅读。
6. 详情页顶部不再出现与审阅工具条重叠的语言 / 主题切换按钮。
7. 替换图片浮层中的当前图片选择，不再被长 narrative 文案撑成长条输入观感。
8. 用户向下滚动时，章节卡片、图片块、文案卡片会以克制的 reveal 节奏进入，而不会完全静态堆叠。
