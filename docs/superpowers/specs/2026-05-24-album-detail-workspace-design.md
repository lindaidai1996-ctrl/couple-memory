## 背景与问题

当前相册详情页是一个长滚动页面，但“照片详情”和“编辑章节”分别通过独立浮层打开。实际使用中，当用户滚动到页面较下方再打开这些入口时，详情层与当前阅读位置之间缺少稳定关系，用户容易感知为：

- 详情层出现在页面顶部附近，不在当前视口语义里
- 长列表上下文被打断，关闭后需要重新找刚刚的照片或章节
- 桌面端和移动端都沿用同一种开层方式，窄屏上的可读性和操作连续性不足

这使相册详情页更像“长列表 + 临时弹窗”，而不是围绕整理任务的稳定工作台。

## 设计目标

1. 让照片详情和章节编辑都与当前相册滚动上下文保持稳定关系。
2. 桌面端优先保证“边看列表边编辑”，移动端优先保证“窄屏可读和可操作”。
3. 把照片详情和章节编辑收敛到同一个详情层骨架里，减少交互分裂。
4. 保留当前相册整理主线，不重写上传、章节创建、多选移动等既有流程。

## 非目标

- 不改写相册详情页的数据接口
- 不新增章节摘要生成逻辑
- 不重做章节创建和批量移动的信息架构
- 不引入新的路由层或详情子页面

## 核心方案

### 1. 相册详情页改为“主内容 + 统一详情工作台”

相册详情页改造成两层响应式结构：

- 桌面端：页面主体保持当前长内容流，右侧新增固定详情工作台
- 移动端：详情工作台改为全屏 `sheet`

主内容继续包含：

- 相册叙事卡
- 上传区
- 章节区
- 其他瞬间区

详情工作台只负责承接：

- 照片详情查看与编辑
- 章节标题 / 背景说明编辑

### 2. 照片详情与章节编辑共用同一套详情层骨架

统一骨架的结构固定为三段：

- 顶部：标题、对象说明、关闭动作
- 中部：独立滚动的详情内容
- 底部：主操作和次操作

同一骨架下根据当前对象切换内容：

- `photo`
  - 大图预览
  - 关键信息摘要
  - 文案与上下文编辑
  - EXIF 信息
  - 同章节照片前后切换
- `chapter`
  - 章节标题
  - 背景说明
  - 章节说明文案

统一骨架的目的不是合并业务字段，而是统一层级、关闭方式、滚动方式和底部动作区。

## 交互规则

### 打开规则

- 点击章节卡里的照片，打开 `photo` 工作台
- 点击“编辑章节”，打开 `chapter` 工作台
- 打开详情时，主内容区保持原滚动位置不变

### 关闭规则

- 桌面端支持右上角关闭和 `Esc`
- 移动端支持头部关闭按钮
- 关闭详情后，不跳回页面顶部

### 编辑规则

- 照片详情默认展示“大图 + 信息 + 编辑”的混合工作区，而不是只给 tab 空白页
- 桌面端在可视区内同时看到预览和表单
- 移动端仍保留信息分段，但头部和底部操作区固定，内容区独立滚动

### 更新规则

- 照片保存后保持当前详情工作台打开，仅刷新当前相册数据
- 章节保存后保持当前详情工作台打开，仅刷新当前相册数据
- 如果当前详情对象在刷新后仍存在，继续停留在该对象

## 响应式策略

### 桌面端

- 页面主结构改成 `content + inspector`
- `inspector` 使用 `sticky`，始终相对于当前 dashboard 内容壳停留在可视范围内
- 当没有选中任何详情对象时，右侧显示空状态说明，告诉用户可以点击照片或章节进入工作台

### 移动端

- 详情工作台以 `fixed inset-0` 全屏 `sheet` 打开
- 顶部标题栏固定
- 底部操作栏固定
- 中间内容独立滚动

## 状态模型

新增页面级统一详情状态：

```ts
type AlbumDetailSurfaceState =
  | {
      kind: 'photo'
      photoId: string
      chapterPhotoIds: string[]
    }
  | {
      kind: 'chapter'
      chapterId: string
    }
  | null
```

规则：

- 页面任何时刻只允许一个详情工作台处于打开状态
- 详情状态只保存实体 `id`，不直接保存快照对象
- 实际渲染对象从最新 `album` 数据中派生，避免保存后详情被关闭或显示旧内容

## 实现边界

### 保留

- `ChapterComposerDrawer`
- `MoveToChapterDialog`
- 现有章节摘要和封面操作接口

### 重构

- 逐步替换 `PhotoDetailModal` 的“独立 modal 外壳”职责
- 替换 `ChapterDetailDrawer` 的“独立 drawer 外壳”职责
- 在相册页引入统一 `workspace surface`

## 验证要求

1. 桌面端打开照片详情时，页面右侧出现工作台，主内容仍停留在原滚动位置。
2. 桌面端打开章节编辑时，不再从页面边缘单独抽出另一个 drawer。
3. 移动端打开照片详情或章节编辑时，都以全屏 `sheet` 展示。
4. 保存照片或章节后，详情工作台不自动关闭。
5. 现有相册页纯函数测试继续通过，并补充统一详情状态相关回归测试。
