# P1 时间线叙事增强设计

## Current Status

- Written on 2026-05-22 to close the last unchecked P1 item: timeline should no longer feel like a simple stack of shooting dates.
- This design keeps the current milestone model and focuses on narrative restructuring in the dashboard timeline experience.

## 1. 目标

让时间线从“按日期堆叠的节点列表”升级成“按阶段阅读的故事线”。

这次增强同时覆盖两层：

- 节点层：单个里程碑要更像可阅读卡片，而不是数据库条目
- 结构层：已确认里程碑要出现阶段感，而不是只有一条线性列表

不在本次范围内的内容：

- 不新增数据库表或持久化的阶段模型
- 不引入新的 AI 生成流程
- 不改动待确认候选区的核心职责

---

## 2. 设计结论

采用“前端即时分段 + 叙事卡片增强”的最小可交付方案。

核心原因：

- 它可以在不改 schema 的前提下完成 P1 收口
- 它能直接复用现有里程碑数据和现有编辑能力
- 它把改动集中在展示层，风险低，回退简单

本次不采用“新增时间线阶段模型”的方案，因为那已经进入新的产品层级，更适合在后续验证通过后再升级。

---

## 3. 用户体验变化

### 3.1 待确认候选区

保持现状，继续承担整理入口职责：

- 展示 AI 候选
- 给出推荐理由
- 支持确认、忽略、编辑
- 给出近日期同地点的重复清理建议

这里不做故事分段，因为用户在这一段的任务是“筛选和确认”，不是“阅读故事”。

### 3.2 已确认时间线

从“平铺节点列表”改成“故事阶段列表”。

每个阶段包含两部分：

- 段首摘要
- 阶段内节点卡片

段首摘要至少展示：

- 时间范围
- 节点数量
- 一句自动生成的阶段说明

阶段说明不追求文学化，目标是帮助用户快速理解“这一段回忆大致在讲什么”。

### 3.3 节点卡片

每个里程碑卡片按阅读顺序重排：

1. 标题
2. 叙事描述
3. 辅助信息（日期、地点、关联照片、AI 标识）

如果节点没有描述，不再直接留空，而是根据已有字段生成一句轻量阅读提示，例如：

- 围绕地点的提示
- 围绕时间节点的提示
- 围绕关联照片的提示

目标不是伪造内容，而是降低“空卡片”的结构噪音。

## 4. 数据与规则

### 4.1 输入数据

沿用当前时间线接口返回字段：

- `id`
- `title`
- `description`
- `date`
- `locationName`
- `isAutoGenerated`
- `photo`

只对已确认里程碑做分段，`isAutoGenerated === true` 的候选仍停留在候选区。

### 4.2 分段规则

分段使用稳定的启发式规则：

- 相邻确认节点的日期跨度超过阈值时切段
- 相邻节点地点相同且日期接近时优先保留在同一段
- 没有地点信息时只按时间连续性判断

初版建议：

- 默认日期跨度阈值为 45 天
- 同地点近邻阈值为 14 天

这两个阈值写成纯函数常量，便于后续调参。

### 4.3 阶段摘要规则

阶段摘要由现有数据即时生成：

- 优先使用组内高频地点
- 其次使用时间范围
- 最后补充节点数量语义

例如：

- “这段回忆主要围绕广州展开，像是一段连续发生的相处片段。”
- “这一段记录了年中几次重要节点，适合作为同一个阶段阅读。”

如果某组只有一个节点，则不强行生成复杂摘要，只给出简短说明。

### 4.4 节点阅读提示规则

当 `description` 为空时，生成非虚构提示文案：

- 有地点和照片：强调“这是与某地点关联的一次具体时刻”
- 只有地点：强调“这是与某地点有关的节点”
- 只有照片：强调“这是从照片里提炼出的一个节点”
- 都没有：强调“这是值得保留在时间线中的一个阶段节点”

这些文案只作为界面阅读辅助，不写回数据库。

## 5. 组件与代码结构

### 5.1 页面层

继续在 [src/app/(dashboard)/timeline/page.tsx](/Users/user/Documents/codes/work/src/app/(dashboard)/timeline/page.tsx) 完成数据拉取与编辑动作，不重构 API 交互。

页面内部结构调整为：

- `pending milestones`
- `confirmed narrative groups`

### 5.2 纯函数层

在时间线页附近提取一组纯函数，负责把里程碑转换成叙事展示数据：

- `buildTimelineNarrativeGroups`
- `buildTimelineNarrativeSummary`
- `buildTimelineReadingMeta`

这些函数的职责分别是：

- 分组
- 生成阶段摘要
- 生成节点阅读辅助文案与展示元信息

这样可以把规则逻辑和 JSX 拆开，便于测试与后续调参。

### 5.3 展示层

新增或内聚两个展示单元：

- `NarrativeGroupSection`
- 增强版 `MilestoneCard`

`NarrativeGroupSection` 负责：

- 渲染段首信息
- 输出组内节点列表

增强版 `MilestoneCard` 继续保留原有编辑、确认、删除动作，但视觉信息层级调整为叙事优先。

## 6. 错误处理与边界

- 当确认区只有 0 个节点时，不渲染阶段区块
- 当确认区只有 1 个节点时，允许降级为“单阶段单节点”
- 当所有节点都缺少地点和描述时，仍能只依赖日期完成分段
- 当日期格式异常时，不新增自定义容错逻辑，沿用当前页面已有行为

不引入新的失败态接口，因为本次不新增网络请求。

---

## 7. 测试策略

这次改动先以纯函数测试为主，再做页面级回归检查。

必须覆盖的测试：

- 日期跨度过大时切成不同阶段
- 同地点且时间接近时保持同一阶段
- 无描述节点会生成阅读辅助提示
- 单节点阶段摘要不会过度包装
- 多节点阶段能产出时间范围和数量信息

验证顺序：

1. 先写失败测试
2. 再补最小实现
3. 运行相关测试
4. 再跑 `eslint`

---

## 8. 验收标准

满足以下条件即可把 P1 最后一项视为收口：

- 用户在已确认时间线里先看到“几个阶段”，再看到“阶段内节点”
- 单个里程碑卡片即使缺少描述，也不会退化成纯元信息条目
- 时间线页的确认区仍保持整理效率，不因叙事增强而变重
- 实现不依赖数据库迁移或新后台模型

---

## 9. 后续升级位

如果这版验证效果稳定，下一阶段可以升级为持久化叙事结构：

- 新增可编辑的阶段标题与阶段说明
- 支持手动调整里程碑归属阶段
- 支持 AI 提供阶段命名候选

这些能力不属于本次 P1 收口范围，但本次纯函数和展示结构应为后续升级保留空间。
